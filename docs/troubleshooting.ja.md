# BC4LinearImport トラブルシューティング

**言語:** [English](troubleshooting.md) | 日本語

このページでは、テクスチャが変換されなかった理由、再インポートで何も見つからなかった理由、そして「それは想定どおりの挙動なのか」を確認しやすくまとめています。

まず普段の使い方を確認したい場合は [使い方ガイド](usage.ja.md) を、短い概要から見たい場合は [README（日本語）](../README.ja.md) を読んでください。

## まずは短く答えると

### なぜ再インポートで `0 eligible textures` と表示されたの？

これは、現在のプロジェクト内で **必要な条件をすべて同時に満たした** テクスチャが 1 つも見つからなかった、という意味です。

確認済みのよくある理由は次のとおりです。

- `Project Settings > BC4 Linear Import` で `Enable BC4 linear import` がオフになっている
- そのテクスチャ、または親フォルダーが `Excluded assets and folders` に入っている
- ファイルが対応している元画像形式ではない（`PNG`、`JPG`、`JPEG` 以外）
- Unity がそのテクスチャを、このワークフローで使う BC4 対象として現在は扱っていない

どこを見ればいいか迷う場合は、[何が BC4 対象として数えられるの？](#何が-bc4-対象として数えられるの) と [除外設定とプロジェクト全体の無効化](#除外設定とプロジェクト全体の無効化) を先に見るのがおすすめです。

### なぜ自分のテクスチャは変換されなかったの？

BC4LinearImport が動くのは、**次の条件がすべて真のときだけ**です。

1. 元ファイルが `PNG`、`JPG`、または `JPEG`
2. `Enable BC4 linear import` が有効
3. アセットパスが除外されていない
4. texture importer が現在 BC4 対象になっている
5. detector が「変換する」と判断し、安全のためにスキップや unknown を返していない

このうち 1 つでも外れると、テクスチャは変わらないことがあります。これはたいていクラッシュではなく、安全側に倒した no-op です。

### Default パネルに `BC4` と見えていれば対象になるの？

いいえ。Default パネルに `BC4` が見えている **だけ** では、このワークフローでは対象扱いになりません。

いちばん安全に確認されているルートは、明示的な **Standalone BC4** 設定です。さらに、条件つきで対象になりうる自動 **Single Channel** パスも観測されていますが、こちらは常に保証されるものとして扱わないでください。

### compute shader が使えないと失敗するの？

変換では、使えるときは compute 経路が優先されますが、**必須条件ではありません**。

compute が使えない、対象テクスチャではサポートされない、または compute 側の試行が失敗した場合でも、ツールは CPU 経路へフォールバックします。つまり、compute を使えなくても変換自体は成功することがあります。

## 対応している元ファイル形式

対応している元ファイル形式は次の 3 つです。

- `PNG`
- `JPG`
- `JPEG`

この対応は元ファイルの拡張子ベースです。元ファイルがそれ以外なら、BC4LinearImport は処理しません。

## 何が BC4 対象として数えられるの？

ざっくり言うと、BC4LinearImport は Unity が Standalone 側で **BC4-targeted** として扱っているテクスチャ向けの機能です。

### いちばん安全に確認されているルート

いちばんわかりやすい対応ケースは次のとおりです。

- Standalone override が有効になっている
- Standalone format が明示的に `BC4` になっている

いちばん予測しやすい結果がほしいなら、このルートを目標にするのが安心です。

### 条件つきで観測されるルート

次の条件では、Unity が自動的に対象として数えるケースも観測されています。

- Standalone override が **無効**
- 自動で決まる Standalone format が `BC4`
- texture type が `Single Channel`
- single-channel component が `Red` または `Alpha`

このルートは便利ですが、環境依存です。**観測された挙動** として扱い、「どの Unity 環境でも必ずそうなる約束」とは考えないでください。

### 対象にならない例

次のケースは、対象外であることが確認されています。

- Default パネルに `BC4` が見えていても、Standalone override が無効
- Standalone override は有効でも、Standalone format が `BC4` **ではない**
- `Single Channel` テクスチャでも、自動 Standalone `BC4` に解決されない

## 何も変わらなかったときによくある理由

テクスチャが変わらない場合は、まず次の確認済みポイントを見てください。

### 1. 元ファイルの拡張子が未対応

BC4LinearImport が対応しているのは、`PNG`、`JPG`、`JPEG` の元ファイルだけです。

拡張子がそれ以外なら、このワークフローは安全に何もしません。

### 2. プロジェクト全体で無効になっている

`Enable BC4 linear import` がオフなら、プロジェクト全体でワークフローが無効になります。

影響するのは次の両方です。

- 通常のインポート時の自動処理
- `Reimport Eligible PNG/JPG/JPEG Textures` または `Tools > BC4 Linear Import > Reimport Eligible Textures` による手動再インポート

### 3. アセットまたはフォルダーが除外されている

そのアセット自身のパスが除外されている場合や、除外フォルダーの中にそのテクスチャがある場合、BC4LinearImport はそのまま触らずに残します。

これも、自動インポート時と手動再インポートの両方に影響します。

### 4. importer が現在 BC4 対象ではない

グレースケール画像であることだけでは足りません。

上で説明した BC4 対象条件を texture importer が満たしていない場合、変換に進む前の段階で止まります。

### 5. detector が安全のため変換しない判断をした

targeting の確認を通ったあとでも、source detector には次の 3 つの結果があります。

- **Convert to linear**: 変換を実行する
- **Skip conversion**: 元画像がすでに linear 前提に見える、またはそのままにすべきと判断する
- **Unknown**: 安全に分類できないので、推測せず何もしない

そのため、いちおう対象条件は満たしていても、no-op になることはありえます。

### 6. 元ファイルのバイト列が取得できない、または安全に読めなかった

detector は、変換すべきかどうかを判断するために元ファイルを読みます。

元ファイルのバイト列が欠けている、空である、途中で切れている、または安全確認に失敗した場合は、無理に推測せずテクスチャをそのまま維持します。

### 7. 自動ルートがその環境では観測されなかった

自動 `Single Channel` ルートに頼っている場合は、条件つきであることを思い出してください。

Unity がそのケースを自動 Standalone `BC4` として解決しない環境では、このワークフローの対象になりません。

## 除外設定とプロジェクト全体の無効化

この 2 つは似ていますが、役割が違います。

- `Enable BC4 linear import`: プロジェクト全体でワークフローをオン / オフする
- `Excluded assets and folders`: ワークフロー自体は有効のまま、指定したアセットやフォルダーだけをスキップする

数枚だけ触られたくない場合は、除外設定を使うのが向いています。

プロジェクト全体で BC4LinearImport を止めたい場合は、プロジェクト全体のトグルを使ってください。

## 診断

条件つきの自動ルートが自分の環境で見えているか確かめたい場合は、次を使います。

- `Tools > BC4 Linear Import > Diagnostics > Observe Targeting`

特に役立つのは、次のような場面です。

- `Single Channel` の設定が、自動 Standalone `BC4` として観測されているか確認したいとき
- 「このテクスチャは BC4 対象のはず」と決めつける前に、もう少し根拠がほしいとき

これは修復ツールではなく、観測ツールです。「この設定はここで BC4-targeted として扱われているのか？」を確かめる助けになります。

## Compute フォールバック

BC4LinearImport は、使えて条件も合う場合には compute ベースの変換経路を優先します。

その経路が使えない、対象テクスチャではサポートされない、または試行中に失敗した場合は、CPU 変換経路へフォールバックします。

つまり、次のように考えて大丈夫です。

- compute が使えなくても、機能そのものの **必須失敗条件** ではない
- CPU 経路で変換が完了することがある
- compute 関連の警告が出ても、それだけで未変換だったとは限らない
